Intelligent House Database Design

1. Introduction

This work is a database design for an Intelligent House system that can control lighting and temperature both automatically and manually. The system collects and manages data generated by sensors (temperature and light sensors) and remote controls. The database design is based on three main components: Room, Sensor, and Remote. The design process involves conceptual modeling using UML, logical modeling, normalization, and implementation using SQL.

2. Conceptual Model

The conceptual model is created using UML class diagrams. The model includes the following classes:

  • Room: Represents the physical rooms in the house, with attributes such as Room_ID, Room_Name, and Surface_Area.
  • Sensor: Represents the sensors (temperature and light sensors) in the house, with attributes such as Sensor_ID, Operating_Status, Sensor_Type, and Unit_of_Measure.
  • Remote: Represents the remote controls used to manage the sensors, with attributes such as Remote_ID, Model_Num, and Purchase_Date.

The relationships between these classes are also defined in the UML diagram. For example, a Remote can control multiple Sensors, and each Sensor is located in a specific Room.

3. Logical Model

The logical model represents the specific nature of attributes, distinguishing between primary keys and foreign keys. The primary key of a class is the unique identifier used to call other functionally dependent attributes. A foreign key is a primary key in another class, used to create associations between classes.

The logical model includes the following tables:

  • Room: (Room_ID, Room_Name, Surface_Area)
  • Location: (Location_ID, Location_Name, Room_ID)
  • Sensor: (Sensor_ID, Operating_Status, Sensor_Type, Unit_of_Measure, Model_Num, Replacement_ID, Location_ID, Remote_ID)
  • Heating_ON_Reading: (Heating_On_Time, Heating_On_Date, Sensor_ID, Heating_On_Value)
  • Heating_OFF_Reading: (Heating_Off_Time, Heating_Off_Date, Sensor_ID, Heating_Off_Value)
  • Lighting_ON_Reading: (Lighting_On_Time, Lighting_On_Date, Sensor_ID, Lighting_On_Value)
  • Lighting_OFF_Reading: (Lighting_Off_Time, Lighting_Off_Date, Sensor_ID, Lighting_Off_Value)
  • Remote: (Remote_ID, Model_Num, Purchase_Date, Mobility, Sensor_ID, Location_ID, Lighting_Start_Time, Lighting_Start_Date, Heating_Start_Time, Heating_End_Time)

4. Normalization

The database is normalized to reduce redundancy and ensure consistency. The normalization process involves identifying functional dependencies and applying normal forms (1NF, 2NF, and 3NF). For example, the Room table is in 1NF because all attributes are atomic, and the Sensor table is in 2NF because all non-key attributes are fully dependent on the primary key.

5. Implementation

The database is implemented using SQL. The following SQL code creates the tables and defines the relationships between them: 

            
                CREATE TABLE Room(
                    Room_ID varchar(255),
                    Room_Name char,
                    Surface_Area int,
                    PRIMARY KEY (Room_ID)
                );

                CREATE TABLE Location(
                    Location_ID varchar,
                    Location_Name char,
                    Room_ID varchar,
                    PRIMARY KEY (Location_ID),
                    FOREIGN KEY (Room_ID) REFERENCES Room(Room_ID)
                );

                CREATE TABLE Sensor(
                    Sensor_ID varchar,
                    Operating_Status char,
                    Sensor_Type varchar,
                    Unit_of_Measure char,
                    Model_Num varchar,
                    Replacement_ID int,
                    Location_ID varchar,
                    Remote_ID varchar,
                    PRIMARY KEY (Sensor_ID),
                    FOREIGN KEY (Location_ID) REFERENCES Location(Location_ID),
                    FOREIGN KEY (Remote_ID) REFERENCES Remote(Remote_ID)
                );

                -- Additional tables and constraints can be added similarly.

6. Testing and Validation

The database was tested by inserting sample data and running queries to ensure that the relationships and constraints work as expected. For example, a query was run to check if a Remote can control multiple Sensors, and another query was run to ensure that each Sensor is located in a specific Room.

7. Future Scope

The design can be further developed by adding more sensors (e.g., humidity sensors, motion sensors) and integrating the system with IoT devices. Additionally, a user interface can be developed to make the system more user-friendly.
